# Аудит производительности и дизайна — Nearbus (розклад)

**Дата:** 2026-07-12
**Объект аудита:** публичная PWA расписания автобусов, обслуживаемая страница `frontend/schedule.html` (rewrite `/` → `/frontend/schedule.html` в `vercel.json`), сопутствующие `tokens.css`, `glass.css`, `sw.js`, `manifest.json`, серверлесс-функции `api/*`, а также `admin.html`.
**Методология:** аудит проведён по фреймворкам из `.agent/skills` — `accessibility-audit` (WCAG 2.2 / POUR), `performance-optimization` (Core Web Vitals), `critique-color`, `critique-information-density`, `design-qa-checklist`, `error-handling-ux`, `loading-states`.
**Важно:** это отчёт только с находками. Никакие файлы сайта не изменялись.

## 0. Резюме

Приложение функционально и визуально аккуратно (единая дизайн-система токенов, «liquid glass», dark mode, staggered-анимации). Однако есть **блокирующие проблемы доступности**, **избыточный рантайм-ререндер**, **утечка секретов и XSS в админке**, а также **мёртвый код на ~150 КБ**.

## 1. Производительность

Целевые метрики (`performance-optimization`): LCP ≤ 2.5 c, INP ≤ 200 мс, CLS ≤ 0.1. Замеры Lighthouse/RUM не выполнялись (нет доступа к продакшн-рантайму) — ниже структурные проблемы, видимые из кода, с ожидаемым влиянием на INP/CLS.

### P1 — Полный ререндер всей ленты по таймеру и на каждый ввод 🟠

`frontend/schedule.html:634-637`
```js
updateClock();
setInterval(updateClock, 10000);
setInterval(render, 60000);   // ← полный rebuild каждую минуту
render();
```
- `render()` (`schedule.html:473-625`) собирает **одну гигантскую строку HTML** для hero + всего таймлайна (до 21 остановки × ~23 чипа времени ≈ 480+ узлов) и присваивает её `heroContent.innerHTML` и `timelineContent.innerHTML`.
- Тот же полный ререндер вызывается на **каждое нажатие клавиши** в поиске: `oninput="onSearch(this.value)"` → `onSearch()` → `render()` (`schedule.html:251, 459-463`). Дебаунса нет.
- **Последствия:** (1) на слабых Android-телефонах ввод в поиске лагает (высокий INP); (2) каждую минуту и на каждый keystroke заново проигрываются входные анимации `.stop-row { animation: sin ... }` со `style="animation-delay:'+(idx*0.025)+'s'"` (`schedule.html:585`) — визуальное мерцание/подёргивание всей ленты без действия пользователя; (3) сбрасывается любое выделение/состояние DOM.
- **Рекомендация:** обновлять только изменяемые части (часы, «следующий» чип, счётчик минут) точечно; поиск — фильтровать через `hidden` без пересборки; авто-обновление — не перестраивать DOM, а пересчитывать классы. Как минимум — дебаунс поиска (~150–200 мс) и снятие анимаций при не-первом рендере.

### P2 — O(n²) и повторный парсинг строк времени 🟡

`schedule.html:571-618` — внутри `filteredStops.forEach` для каждой остановки вычисляется:
```js
var isNextStop = ... idx===filteredStops.findIndex(function(s2){var t2=getTimes(s2.k); return findNext(t2)>=0;});
```
`findIndex` сам сканирует массив и для каждого элемента снова зовёт `getTimes()` + `findNext()` → квадратичная сложность на каждый рендер. Плюс `parseT()` (`schedule.html:423`) заново парсит строки `"H:MM"` через `split`/`parseInt` на каждом кадре, хотя времена статичны. На текущем объёме (21 остановка) не критично, но это лишняя работа на каждый keystroke/минуту. **Рекомендация:** вычислить индекс «следующей» остановки один раз до цикла; кэшировать распарсенные минуты.

### P3 — Мёртвый код ~150 КБ 🟡

- `grep` по всем HTML: **ни `schedule.html`, ни `admin.html` не подключают `app.js` и `routes.js`**.
- `frontend/app.js` — 48 КБ / 1276 строк, `frontend/routes.js` — 103 КБ / 6171 строка. Это, судя по всему, легаси карточной/картографической версии. Файлы лежат в репозитории и деплоятся (rewrite `/((?!api/).*)` отдаёт их), увеличивая вес деплоя и создавая путаницу «где логика».
- **Рекомендация:** удалить, если не используются, либо переместить в `legacy/` вне публичной раздачи.

### P4 — Загрузка шрифта Inter в 5 начертаний 🟡

`schedule.html:11`
```html
<link rel="stylesheet" href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700;800&display=swap"/>
```
Рендер-блокирующий внешний стиль + 5 весов (400–800). `preconnect` есть (хорошо), `&display=swap` есть (хорошо), но фактически используются не все веса, а система уже имеет `-apple-system` фолбэк. **Рекомендация:** сократить набор весов (например 400/600/700), рассмотреть self-host / `preload` основного файла, чтобы убрать зависимость от стороннего origin на критическом пути LCP.

### P5 — Сетевые пинги аналитики каждые 30 c 🟡

`schedule.html:643-665` — `setInterval(sendPing, 30000)` + пинг на каждый `visibilitychange`. Постоянный сетевой трафик и апсерт в БД на каждого посетителя. Расход батареи/трафика на мобильных, нагрузка на `api/ping`. **Рекомендация:** увеличить интервал / использовать `navigator.sendBeacon`, объединять события.

### P6 — Расписание зашито в HTML

Объект `SCH` (`schedule.html:311-338`) — это ~35 КБ данных прямо в HTML (итоговый файл 57 КБ). Любое изменение расписания инвалидирует кэш всей страницы; данные нельзя закэшировать отдельно. Приемлемо для хакатона, но для роста — вынести в отдельный `schedule.json` с собственной политикой кэша.

## 2. Доступность (WCAG 2.2 / POUR)

### A1 — Запрещён зум страницы 🔴 Critical (WCAG 1.4.4 Resize Text, 1.4.10)

`schedule.html:5`
```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover"/>
```
`maximum-scale=1.0, user-scalable=no` блокирует pinch-to-zoom. Для приложения общественного транспорта, которым пользуются в т.ч. пожилые и слабовидящие, это прямой барьер. **Fix:** убрать `maximum-scale` и `user-scalable=no`.

### A2 — Провал контраста третичного текста 🟠 Major (WCAG 1.4.3)

`tokens.css:28` — `--text-tertiary: var(--gray-400) /* #aeaeb2 */` на `--bg: #f2f2f7` (светлая тема) даёт контраст **≈1.9:1** при требуемых 4.5:1. Используется в: плейсхолдере поиска (`::placeholder`, `schedule.html:41`), «Дані відсутні» (`:614`), стрелке направления hero (`:71`), тексте `.no-results` (`:90`). Плюс `.time-chip--past { opacity: 0.35 }` (`:116`) поверх уже вторичного цвета — итог ~1.7:1. **Fix:** для третичного текста на светлой теме взять минимум `gray-500 #8e8e93` (и проверить — он даёт ~3:1, для мелкого текста всё ещё мало; лучше `gray-600`).

### A3 — Нет видимого фокуса клавиатуры 🟠 Major (WCAG 2.4.7)

У интерактивных элементов заданы только `:active` состояния (`.back-btn:active`, `.segment__btn`, `.dir-btn:active`, `.hero__day-btn:active`, `.stop-row__mystop-btn`, `.settings-dropdown__item:active`). `:focus-visible` не определён нигде, кроме `.search-input:focus`. При навигации с клавиатуры/свитча пользователь не видит, где находится. **Fix:** добавить единый `:focus-visible { outline: 2px solid var(--accent); outline-offset: 2px }`.

### A4 — Тач-таргеты меньше 44×44px 🟠 Major (WCAG 2.5.8, `design-qa-checklist`)

- `.stop-row__mystop-btn` — 28×28 (`:110`)
- кнопка смены направления в hero — `width:30px;height:30px` (`:512`)
- `.search-clear` — 20×20 (`:42`)
- `.dir-btn` — 42×42 (близко, но < 44)
- `#settingsBtn` — иконка 20 + padding 8 ≈ 36px (`:195`)

**Fix:** довести кликабельную область до 44px (можно через увеличенную зону нажатия/`padding`, сохранив визуальный размер).

### A5 — Неполная семантика табов и ARIA

- Табы `role="tab"`/`role="tablist"` (`:218-239`) без связанных `role="tabpanel"`/`aria-controls` и без обработки стрелок клавиатуры (ARIA Authoring Practices требует Left/Right). Фактически это переключатели — возможно, роль `tab` вообще избыточна.
- Живые часы в hero (`#liveClock`) и авто-обновляющийся таймлайн меняются без `aria-live` — скринридер не знает об обновлении.
- Модалка отзыва (`:181-190`) не имеет `role="dialog"`, `aria-modal`, ловушки фокуса и возврата фокуса; закрывается только кликом по фону/кнопке (нет Esc).

### A6 — Иконки-кнопки: проверить озвучку

`aria-label` присутствует у settings/direction/search — хорошо. Но кнопки-иконки без текста (mystop) полагаются на `title` + `aria-pressed` — `title` не всегда озвучивается; добавить `aria-label`.

## 3. Цвет и визуальный язык (`critique-color`)

- **Контраст:** см. A2 — `major issue`. Тёмная тема в целом ок (`--text-tertiary #8e8e93` на `#000` ≈ 4.6:1), проблема именно светлой темы.
- **Семантика цветом без дублирующего признака:** статус «наступний рейс» у `.time-chip--next` и `.stop-row--next-stop` кодируется преимущественно синим фоном/цветом; «моя зупинка» — оранжевым. Для дальтоников (дейтер/протанопия) синий-vs-серый и оранжевый-vs-серый различимы слабо. Иконка/бейдж «годинник» рядом с ближайшим временем частично спасает, но статус остановки в ленте — фактически только цвет. **Fix:** добавить текстовую метку/иконку к состоянию, не только цвет.
- **Согласованность палитры:** хорошо — почти всё через токены. Точечные одноразовые значения в инлайне: `#1c88ff`/`#006ce0` в `.map-cta` (`:139`), множество `rgba(255,149,0,...)` вместо токена `--warning`, `rgba(0,122,255,...)` вместо `--accent`. Мелкий tech-debt.
- **Манифест:** `theme_color: #f2f2f7` (светлый) при `background_color: #000000` (чёрный) — несогласованно; на тёмной теме верхняя панель браузера будет светлой. В HTML нет `<meta name="theme-color">` вовсе (0 вхождений).

## 4. Информационная плотность (`critique-information-density`)

### I1 — «Стена времён» 🟠 Major

`schedule.html:602-612` — для **каждой** остановки рендерятся **все** времена на день (20+ чипов). При 21 остановке это 400+ числовых чипов на одном экране (`major issue` по когнитивной нагрузке и приоритету контента). Пользователю на остановке нужно в первую очередь «когда следующий отсюда», а не полный график всех остановок сразу.

**Fix (progressive disclosure):** по умолчанию показывать 2–3 ближайших времени на остановку + «ещё N» для раскрытия; полный список — по тапу/в детальном виде. Hero это частично решает для «моей остановки», но лента дублирует всё.

### I2 — Приоритет контента

Hero-блок хорошо выделяет главное (следующее отправление). Но `stats-bar` («Інтервал», «Рейсів ще»), баннер направления, две сегмент-группы (маршрут + день) и поиск — всё конкурирует над лентой. На маленьком экране первичное действие («когда автобус») оттесняется вниз. **Fix:** пересмотреть вертикальный порядок, свернуть вторичное.

## 5. Дизайн / UX и оформление

### D1 — Ложный аффорданс кликабельности 🟡
`.stop-row { cursor: pointer }` (`:101`), но в `render()` у строки **нет** обработчика клика (`:585`) — кликается только вложенная кнопка «моя зупинка». Курсор-рука обещает действие, которого нет. **Fix:** либо сделать строку кликабельной (раскрытие/выбор), либо убрать `cursor:pointer` и `user-select:none`.

### D2 — Сверхмелкая типографика 🟠
9px: `.hero__day-label`, `.hero__dir-from/to` (`:66,70,72`); 10px: `.hero__stop-label`, `.stat-chip__label` (`:54,81`); 11px: `.section-header__title`, бейджи (`:86,87,109`). 9px uppercase с трекингом на мобильном — на грани читаемости, особенно для транзитной аудитории. **Fix:** поднять минимум до 11–12px, uppercase-лейблы не мельче 11px.

### D3 — `alert()` для отзыва 🟡
`schedule.html:412,414` — успех/ошибка отправки через нативный `alert()`. Выбивается из «liquid glass» эстетики, блокирует поток, невозможно стилизовать. **Fix:** тост/инлайн-статус в модалке (см. `loading-states`/`error-handling-ux`).

### D4 — Обработка ошибок и состояния загрузки
- Нет skeleton/loading для первого рендера (данные инлайн, поэтому быстро — приемлемо).
- `error-handling-ux`: ошибка отправки отзыва — общий `alert('Помилка...')` без ретрая. Оффлайн-состояние приложения (нет сети) визуально не индицируется, хотя это транспортное PWA, часто используемое при плохой связи.
- Пустое состояние поиска сделано хорошо (`.no-results`, `:560-567`) — плюс.
- Несогласованность иконографики: везде line-SVG, но в пустом состоянии — эмодзи `🔍` (`:563`). `critique-brand-consistency`: заменить на SVG из общей системы.

### D5 — Именование/структура
`.back-btn` переиспользована как кнопка настроек-шестерёнки (`:195`) — семантически вводит в заблуждение (это не «назад»). Заголовок всегда «Розклад» без навигации назад. Тяжёлое смешение инлайн-стилей и классов (много `style="..."` в разметке и в JS-строках) усложняет поддержку — расходится с наличием аккуратной токен-системы.

### D6 — Производительность отрисовки эффектов
14 использований `backdrop-filter: blur(20–30px)` в инлайн-CSS + `glass`-тени. Стек полупрозрачных размытий на слабых GPU даёт просадки при скролле/анимациях. Плюс на браузерах без поддержки `backdrop-filter` элементы становятся полупрозрачными с низким контрастом (см. A2). **Fix:** ограничить количество одновременных blur-слоёв, задать непрозрачный фолбэк через `@supports not (backdrop-filter: blur())`.

### D7 — `prefers-reduced-motion`
Медиазапрос есть (`:163-170`) и покрывает основные анимации — хорошо. Не покрыты: `.map-cta` transitions, `.stop-row__*` transform-переходы, `pdot`-пульс покрыт, `chip-pulse` покрыт. В целом near-pass, стоит добить остаток.

## 6. PWA / Service Worker

### W1 — Network-first для оболочки 🟡
`sw.js:73-92` — оболочка (`schedule.html`, CSS) грузится network-first (по коммиту — намеренно, «always load fresh schedule»). Это теряет ключевое преимущество PWA — мгновенную загрузку из кэша; при медленной сети старт медленный (хотя есть оффлайн-фолбэк). Компромисс «свежесть vs скорость» решён в пользу свежести. **Рекомендация:** stale-while-revalidate для оболочки + отдельный версионируемый `schedule.json` network-first — тогда и быстро, и данные свежие.

### W2 — Кэш `style.css`, не нужный основному приложению 🟡
`sw.js:8-14` прекэширует `style.css`, но `schedule.html` его **не подключает** (нужен только `admin.html`). Лишние 36 КБ в прекэше оболочки. И наоборот — `glass.css`, который **используется** главной страницей, в `SHELL_FILES` отсутствует.

### W3 — Манифест 🟡
- `icons[].purpose: "any maskable"` (`manifest.json:16,22`) — совмещение ролей: maskable-иконка обрезается safe-zone (10% паддинг), поэтому «any» будет выглядеть урезанной. Лучше отдать раздельные иконки `any` и `maskable`.
- Нет `<link rel="apple-touch-icon">` в `schedule.html` (0 вхождений) — на iOS ярлык на домашний экран возьмёт скриншот вместо иконки.
- Нет `<meta name="description">` и Open Graph (`:head`) — плохой шеринг/SEO для публичной страницы.

## 7. Безопасность (обнаружено попутно — критично)

### S1 — Секреты в исходном коде 🔴 Critical
- Пароль админки в открытом виде: `api/stats.js:16` → `req.query.password !== 'nearbus2024'`. Пароль передаётся **в query-строке** (`admin.html:70`, `getFeedback` аналогично) — попадает в логи сервера/прокси, историю, реферер.
- Supabase URL и ключ захардкожены как фолбэки во **всех** функциях: `api/feedback.js:1-2`, `api/ping.js:1-2`, `api/stats.js:1-2` (ключ прямо в строке, не только из `process.env`).
- **Fix:** пароль/ключи — только через переменные окружения Vercel; проверку пароля — через заголовок/POST-тело, а не query; ротировать утёкший ключ; для админки — нормальная аутентификация.

### S2 — Stored XSS в админке 🔴 Critical
`admin.html:106-113` — отзывы рендерятся через шаблонную строку в `innerHTML` без экранирования:
