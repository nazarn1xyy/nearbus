# Автобуси Могилева-Подільського — навчальний проєкт

Простий pet-проєкт: WebSocket-сервер симулює рух міських автобусів
по кількох маршрутах, а фронтенд показує це на карті в реальному часі.

**GPS немає** — рух автобусів повністю імітується на бекенді: координати
зупинок задані вручну, а між зупинками сервер генерує проміжні точки
і "веде" автобус по них по колу.

Бекенд є у двох варіантах, з однаковою логікою — обери, який зручніше:
- `backend/` — на Node.js (`ws`)
- `backend-python/` — на Python (`asyncio` + `websockets`)

Формат повідомлень в обох однаковий, тому фронтенд для них спільний
і його міняти не треба.

## Структура проєкту

```
mogyliv-bus-tracker/
  backend/                  - варіант на Node.js
    server.js                - WebSocket сервер, головний файл
    simulation.js             - логіка руху автобусів (клас Bus)
    routes.js                 - дані маршрутів і зупинок
    geo.js                    - допоміжні функції (відстань, інтерполяція)
    package.json
  backend-python/            - варіант на Python
    server.py                 - WebSocket сервер, головний файл
    simulation.py              - логіка руху автобусів (клас Bus)
    routes.py                  - дані маршрутів і зупинок
    geo.py                     - допоміжні функції (відстань, інтерполяція)
    requirements.txt
  frontend/
    index.html
    style.css
    app.js                    - підключення до WebSocket + робота з картою Leaflet
```

## Як запустити

### 1. Бекенд

Node.js-варіант:
```
cd backend
npm install
npm start
```

Python-варіант (потрібен Python 3.8+):
```
cd backend-python
pip install -r requirements.txt
python3 server.py
```

Будь-який з них піднімає сервер на `ws://localhost:3001`.
Запускати треба лише ОДИН з двох бекендів, не обидва разом
(інакше другий не зможе зайняти той самий порт).

### 2. Фронтенд

Фронтенд — це звичайні статичні файли (HTML/CSS/JS), окремий сервер
для них не обов'язковий.

**Найпростіше:** просто відкрий `frontend/index.html` подвійним
кліком у браузері (Chrome, Firefox і тд). Це працює, бо сторінка
підключається до бекенду напряму через WebSocket
(`ws://localhost:3001`), а це не залежить від того, як саме
відкрита сама сторінка — хоч через `file://`, хоч через сервер.
Головне, щоб бекенд вже був запущений.

`npx serve` чи будь-який інший локальний сервер — це просто
альтернатива, не обов'язкова вимога. Вона може знадобитись хіба що
якщо у твоєму браузері ввімкнені якісь особливо суворі налаштування
безпеки, які блокують `file://` сторінки, але для звичайного Chrome/
Firefox це не проблема.

## Як це працює

1. Бекенд при старті створює по 2 автобуси на кожен з 3 маршрутів.
2. Кожну секунду (`TICK_INTERVAL_MS` в `server.js`) сервер пересуває
   кожен автобус на один крок вперед по заздалегідь розрахованому
   шляху і розсилає всім підключеним клієнтам нові координати.
3. Фронтенд при підключенні отримує список маршрутів і зупинок
   (`type: "init"`) і малює лінії маршрутів та зупинки на карті.
4. Далі кожне повідомлення `type: "update"` містить нові координати
   автобусів — фронтенд просто пересуває відповідні маркери.
5. Час прибуття (ETA) рахується на бекенді як
   `відстань до наступної зупинки / середня швидкість автобуса`.
   Це груба оцінка, а не точний розрахунок — для навчального проєкту
   цього достатньо.

## Що можна покращити далі (якщо захочеться прокачати проєкт)

- Замінити лінійну інтерполяцію координат на реальний роутинг по
  вулицях (наприклад через OSRM), щоб автобуси їхали точно по дорогах.
- Додати збереження історії руху в базу даних.
- Додати реальний розклад і порівняння "план vs факт".
- Покрити simulation.js/geo.js юніт-тестами.
